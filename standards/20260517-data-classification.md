# Data Classification — AC Billing System

**Status:** binding · 2026-05-17. Every new column or persisted artifact must be classified at PR time; CI guards check.
**Pairs with:** `.ai/standards/20260517-threat-model.md`, `.ai/standards/20260517-CONVENTIONS.md` §7 (logging), ADR 005 (KMS), ADR 004 (tenancy), `.cursorrules` §"Data Privacy & Security".

This file maps every data category we hold to: where it can live, how it must be encrypted, who can see it in logs/backups, and what the breach playbook is.

---

## 1. Classes

| Class | Description | Examples (this system) |
|-------|-------------|------------------------|
| **public** | No confidentiality requirement. Safe to log, share, embed in error responses. | Document type enums, currency codes, CABYS code text, FX rate values. |
| **operational** | Internal identifiers + system state. Not regulated; leak is annoying but not material. | UUIDs, queue task IDs, correlation IDs, state-machine state names, SHA-256 hashes. |
| **financial** | Numbers and structured documents that describe trade between named parties. Regulated under tax law; leak is reputational and contractual harm. | Invoice line amounts, totals, signed XML bodies (which embed line items), Hacienda responses. |
| **pii** | Personally identifying information about natural persons. Regulated under CR Law 8968 (Protección de la Persona frente al tratamiento de sus datos personales). | Customer names, email addresses, phone numbers, physical addresses. |
| **tax_id** | National identifiers (cédula física, jurídica, NITE, DIMEX, Extranjero No Domiciliado, No Contribuyente). Treated as a distinct subclass of PII for finer-grained controls. | All `numeroIdentificacion` fields in our schema. |
| **credential** | Material that authenticates or authorizes something. Leak = direct compromise. | Hacienda OAuth usernames + passwords, signing P12 + PIN, ATV cryptographic keys + PIN, callback tokens, JWT access tokens, app session cookies, Postgres roles. |

Classes are **not** mutually exclusive. A signed XML body is `financial` AND embeds `pii` AND embeds `tax_id`. The strictest class controls how it is handled.

## 2. Handling matrix

| Capability | public | operational | financial | pii | tax_id | credential |
|------------|--------|-------------|-----------|-----|--------|------------|
| Allowed in `print()` (dev) | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ |
| Allowed in `structlog` event line | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ |
| Allowed in HTTP error body to non-tenant caller | ✓ | UUIDs only | ✗ | ✗ | ✗ | ✗ |
| Allowed in HTTP error body to tenant caller | ✓ | ✓ | summary only | ✗ | masked | ✗ |
| Allowed in OpenTelemetry span attributes | ✓ | ✓ | ✗ | ✗ | masked | ✗ |
| Allowed in DB column (plaintext at rest) | ✓ | ✓ | ✓ (in TDE-encrypted RDS) | ✓ (in TDE-encrypted RDS, RLS-scoped) | ✓ (same) | ✗ (must be in Secrets Manager / S3 with KMS envelope encryption) |
| Allowed in S3 object body | ✓ | ✓ | ✓ (KMS CMK per tenant, Object Lock for fiscal artifacts) | ✓ | ✓ | ✓ (separate bucket; see ADR 005) |
| Allowed in backups | ✓ | ✓ | ✓ (encrypted) | ✓ (encrypted) | ✓ (encrypted) | refs only — backups carry the KMS-encrypted blob, never the plaintext |
| Allowed in non-prod env data | ✓ | ✓ | synthetic only | synthetic only | synthetic only | synthetic only |
| Allowed in CI fixtures | ✓ | ✓ | synthetic only | synthetic only | synthetic only | synthetic only |

Definitions:

- **summary only** for `financial`: total amount and currency are OK; per-line breakdown is not.
- **masked** for `tax_id` in error bodies/spans: show only the last 4 characters, prefix with `***`.
- **synthetic** means generated by `apis/tests/fixtures/factories.py`, deterministic, never derived from real customer data.

#### 3.3.1 `payload` field rules per event type

- `Drafted`: `{commercial_document_id, doc_type}` — operational.
- `ValidationFailed`: `{rules_failed: ["R1", ...]}` — operational.
- `Signed`: `{signed_artifact_sha256, key_id, algorithm}` — operational.
- `SignatureFailed`: `{error_code, library}` — operational.
- `ContingencyQueued`: `{reason_ref}` — operational.
- `Submitted`: `{location_url, request_id}` — operational.
- `Received`: `{ind_estado}` — operational.
- `Accepted`/`Rejected`: `{ind_estado, hacienda_response_sha256, reason_codes?}` — operational; `reason_codes` are Hacienda enums, public.
- `Errored`: `{x_error_cause, http_status, validation_exception}` — `x_error_cause` is operational (Hacienda's pre-redacted text, no PII in observed samples; if a future Hacienda change leaks PII into `X-Error-Cause`, the redact helper catches it before persistence).
- `Cancelled`: `{actor_id, reason}` — `reason` is free-text from operator and is treated as **pii** to be safe (cannot persist without redaction or hashing).

## 4. Logging rules (binding)

- Log lines may contain **public** and **operational** values freely.
- Log lines must never contain **financial**, **pii**, **tax_id**, or **credential** values in any form, including:
  - Field names that imply the value (e.g. `password=*` is forbidden because grep tooling can't be sure it's masked).
  - Stack traces that include local variables — set `structlog`'s `include_locals=False` in production.
  - HTTP request/response bodies. Use a registered hook that strips bodies for any path matching `^/(fiscal|master-data|hacienda)`.
- CI greps for these high-risk patterns across `apis/src/` (regex examples; tune to taste):
  - `logger\..*(emisor|receptor)\.(nombre|email|telefono)`
  - `logger\..*identificaci[oó]n`
  - `logger\..*\bclave\b.*=` (allows logging the clave but only when it is a top-level key, never an embedded value)
  - `logger\..*password|secret|token|bearer`
- The helper `acb_platform.logging.redact_pii()` (`apis/src/acb_platform/logging.py`) returns a structurally identical object with all PII / tax_id / credential fields replaced by `[REDACTED]`. Use it on the rare occasion that a structured payload must be logged.

## 5. Backup and restore rules

- **DB backups (RDS automated + cross-region snapshot):** encrypted with the AWS KMS account default CMK; restore role lives in a separate AWS account with its own audit trail.
- **S3 signed-artifact bucket:** versioned, Object Lock in compliance mode for the legal retention window (default 5 years; pin to CPA confirmation when ADR 009 closes). Cross-region replication to a bucket in a separate account.
- **S3 signing-material bucket:** versioned (no Object Lock so legitimate rotations work); KMS CMK per tenant; lifecycle = transition to Glacier Instant Retrieval after 30 d unused.
- Restore tooling **refuses** to write into a target tenant whose `tenant_id` mismatches the manifest's `tenant_id` (per threat model B9).
- Pre-prod restore is permitted **only** with the obfuscation pipeline applied (`.obfuscation/sanitize_data.py` per `.cursorrules`). A non-sanitized prod snapshot cannot be loaded into `staging` or below by tooling.

## 6. Sensitive-data lifecycle hooks

| Stage | Required action |
|-------|-----------------|
| Ingestion (user upload, API write) | Classify on the way in; reject if a `credential` value would land in a non-credential store. |
| Display (dashboard, PDF, email) | Apply per-role masking; tenant role sees full data, support role sees masked. |
| Export (operator-downloaded CSV/JSON) | Header includes class labels per column; export role must hold `tenant.data.export` and step-up auth. |
| Deletion (tenant offboarding) | Crypto-shred: schedule revocation of the tenant CMK with a 30-day waiting period; after revocation, all KMS-encrypted material (including backups) is mathematically unreadable. |
| Right-to-erasure (CR Law 8968 art. 11) | Targeted erasure procedure documented separately (TODO under `.work/plans/operations/`); the underlying primitives (tenant scoping, per-tenant CMK, audit log) make this feasible. |

## 7. Incident playbook (data exposure suspected)

1. Page the security on-call (rotation TBD; placeholder = tech lead).
2. Tag the affected tenant(s) in the incident channel; freeze relevant credentials (rotate `callback_token`, ATV password, signing material).
3. Pull `audit_log` for the affected window; cross-reference with `fiscal_events` and CloudTrail KMS events.
4. Determine class of exposed data per §1. If `credential` or `tax_id` is in scope, treat as Tier 1 incident.
5. Notify the tenant per the DPA; notify Costa Rica regulators (CR Law 8968 art. 28) if required — CPA + legal interpret the threshold.
6. Post-mortem within 5 business days; update threat model and this document.
